      Proyecto Lombok

      Librería para condimentar Java

      1. Descripción

      Lombok es una librería que todo desarrollador Java debería incorporar a sus proyectos desde el primer momento. El objetivo principal del proyecto Lombok es el de evitar tener que escribir código repetitivo.

      Ya nunca más tendremos que escribir los método get o set, los método equals o hashcode, sobrescribir el método toString, podremos automatizar nuestras variables de tipo logging, y mucho más. Y todo esto con una sola anotación.

      Java es un gran lenguaje, pero en ocasiones es muy minucioso con el código en tareas habituales o en el cumplimiento de ciertas prácticas de su infraestructura. Estas, a menudo, no dan ningún valor añadido a al parte de negocio de nuestros programas. Es aquí donde Lombok nos facilitará nuestro trabajo y nos permitirá ser más productivos.

      Su forma de funcionamiento es la de una extensión que se conecta con el proceso de construcción y autogenera código binario Java en los archivos .class, dependiendo de una serie de anotaciones de este proyecto que iremos introduciendo en nuestro código.

      Nota

      Según la página oficial del proyecto Lombok, es una librería java que automáticamente se conecta con nuestro editor y herramientas de construcción para ponerle pimienta a nuestro Java (quizás de aquí el logotipo elegido para este proyecto).

      2. Descarga e instalación de Lombok

      Incluirlo en nuestros desarrollos es sencillo, y su página oficial dispone de instrucciones detalladas dependiendo del sistema usado.

      Me voy a centrar en una instalación común, en un proyecto Maven y en un entorno de desarrollo integrado común, en este caso, Spring Tool Suite 4 para Eclipse. Se puede consultar la página oficial para otras opciones de herramientas de construcción (Javac, Gradle, Ant) y entornos de desarrollo (Eclipse, IntelliJ IDEA, Netbeans, Visual Studio Code, Jboss, MyEclipse) o plataformas (Android, GWT).

      2.1. Descarga

      Podremos descargar la última versión desde la página oficial. Se descargara la librería con el nombre lombok.jar. En este caso la instalación la podemos hacer desde nuestra carpeta habitual de descargas.

      Este archivo también se puede descargar del repositorio central de Maven, aunque en este caso el nombre incluirá el número de versión, por ejemplo lombok-1.18.8.jar, pero se trata del mismo archivo. En este caso la instalación la podemos hacer desde nuestra carpeta habitual de Maven.

      2.2. Instalación en Spring Tool Suite 4 para Eclipse

      3. Un ejemplo básico

      Antes de entrar a a analizar con detalle cada una de las anotaciones que nos brinda lombok, he preferido comenzar con un sencillo ejemplo para tener una visión general de las ventajas que nos aporta el proyecto.

      Encapsular las propiedades de nuestros objetos con métodos getter y setter es una práctica tan común en el mundo Java, y hay tantos frameworks que dependen del patrón Java Bean (una clase con un constructor vacío y métodos get/set para sus propiedades), que la mayoría de los IDEs soportan generación de código par estos patrones.

      Sin embargo, este código tiene el inconveniente de tener que ser continuamente mantenido cuando se crea o renombra una nueva propiedad.

      Aunque voy a comenzar con una entidad JPA de ejemplo con (con su anotación @Entity) para demostrar que las anotaciones de lombok son compatibles con otras anotaciones de otros frameworks, en secciones posteriores me limitaré solo a exponer únicamente las anotaciones de lombok para facilitar las explicaciones.

      Al indicarle las anotaciones @Getter y @Setter le decimos a lombok que genere estos métodos por nosotros para todas las propiedades de la clase. Con la anotación @NoArgsConstructor, generará un constructor vacío.

      Para esta clase de sólo tres propiedades, el ahorro en código ya es significante. Si más adelante añadimos otras propiedades a la clase sucederá lo mismo, ya que al aplicar las anotaciones a la clase, afectará a todas sus propiedades por defecto.

      Podremos afinar más con las anotaciones usando un conjunto de valores de configuración de las anotaciones y aplicándoles directamente a nuestras propiedades en lugar de a la clase, o a ambas. Por ejemplo, si quisiéramos establecer a la propiedad id una visibilidad package o protected por que queremos que sea de lectura pero no de escritura para el código de la aplicación, podríamos hacer esto :

      private @Id @Setter(AccessLevel.PROTECTED) Long id;

      4. Anotaciones de uso más común de Lombok

      Vamos hacer un repaso de las anotaciones más comunes y de todas sus opciones de configuración.

      4.1 @Getter y @Setter

      4.1.1. Resumen

      Cuando anotamos una propiedad con @Getter y/o @Setter, lombok genera automáticamente los métodos get/set por defecto.

      Un método get por defecto simplemente devolvería la propiedad, y para una propiedad como por ejemplo edad lo nombraría como getEdad , o si fuera de tipo booleano, como isEdad.

      Para un método set por defecto lo nombraría como setEdad , devolvería void, y tomaría 1 parámetro del mismo tipo que la propiedad. Simplemente establece la propiedad a este valor.

      El método get/set generado será public a menos que especifiquemos de forma explícita un nivel de acceso con AccessLevel, como se mostrará en el ejemplo siguiente. Los posibles niveles de acceso válidos son PUBLIC, PROTECTED, PACKAGE y PRIVATE.

      También podemos poner las anotaciones @Getter y/o @Setter sobre la clase. Este caso sería equivalente a anotar todos los campos no estáticos de la clase con la anotación correspondiente.

      El texto javadoc sobre la propiedad se copiará para generar los métodos get y set. Por regla general, se copia todo el texto, y el @return se mueve al get, mientras que la líneas @param se mueven al set. Por “mover” se entiende eliminarlo del del javadoc de la propiedad.

      También es posible definir un texto único para cada get/set. Para hacer esto, podemos crear una sección llamada GETTER y/o SETTER. Una sección es una línea del javadoc que contiene 2 o más guiones, y luego el texto GETTER o SETTER, seguido de 2 o más guiones, y nada más en la línea. Si se usan secciones, el @return o las lineas @param se mueven directamente al get o set correspondiente.

      Con Lombok

      import lombok.AccessLevel;
import lombok.Getter;
import lombok.Setter;

public class Persona {
  /**
   * Edad de la persona.
   * 
   * @param edad Nuevo valor para la edad de la persona.
   * @return El valor actual de la edad de esta persona.
   */
  @Getter @Setter private int edad = 10;
  
  /**
   * Nombre de la persona.
   * -- SETTER --
   * Modifica el nombre de esta persona.
   * 
   * @param nombre El nuevo valor.
   */
  @Setter(AccessLevel.PROTECTED) private String nombre;
  
  @Override public String toString() {
    return String.format("%s (edad: %d)", nombre, edad);
  }
}

      4.1.2. Otros puntos a tener en cuenta

      • Para generar los nombres de los métodos, el primer caracter de la propiedad, si es minúscula, se cambia al formato title-cased, en caso contrario, no se modifica. Luego se le pone un prefijo get/set/is.

      • No se genera ningún método si ya existe uno con el mismo nombre y el mismo número de parámetros. No se tienen en cuenta minúsculas o mayúsculas (case insensitive). Por ejemplo, getNombre() no se genera si ya existe un método getNombre(String... x) aunque técnicamente sea posible tener ese método. Esta salvedad existe para evitar confusión. Si por esta razón no se genera el método, se emite una advertencia. Los parámetros de tipo varargs, cuentan como de 0 a N. Podemos marcar un método que pueda entrar en conflicto con otro con la anotación @lombok.experimental.Tolerate para ocultárselo a lombok, y que se genere el método que nos interese.

      • Para propiedades de tipo boolean que comienzan con el prefijo is, seguido inmediatamente de una letra en formato title-case, no se le añade ningún prefijo para generar el nombre del método get.

      • Cualquier variación del tipo boolean no darán como resultado el uso del prefijo is en lugar del prefijo get; por ejemplo, devolver un java.lang.Boolean resulta en un prefijo get, no en un prefijo is.

      • Si hay una anotación en la propiedad que indica que debe ser un campo no nulo, de algunas librerías populares como javax.annotation.Nonnull, resulta en una comprobación explícita del valor null en el set generado. Estas anotaciones se copian automáticamente al lugar correcto, es decir, al método para el get, o al parámetro para set.

      • Para otras anotaciones adicionales, podemos especificar que sean copiadas a través de la propiedad de configuración lombok.copyableAnnotations; más adelante se explicará qué es la configuración de lombok.

      • Se puede anotar una clase con la anotación @Getter/@Setter. Esto es equivalente a anotar todas las propiedades no estáticas de esa clase con esa anotación. Las anotaciones @Getter/@Setter en una propiedad toma precedencia sobre las de la clase.

      • Si se usa el nivel de acceso AccessLevel.NONE simplemente no se generá nada. Sólo es de utilidad en combinación con @Data o un @Getter/@Setter a nivel de clase.

      • @Getter también se puede usar con enums, pero @Setter no , por la razón que los métodos set en las enumeraciones son una muy mala idea.

      4.1.3. Configuración soportada

      Clave / Valor Detalles
      lombok.accessors.chain = [true| false] (defecto: false) Si se establece a true, los set que se generen devolverán this (en lugar de void). *
      lombok.accessors.chain = [true | false] (defecto: false) Si se establece a true, los set que se generen devolverán this (en lugar de void). *
      lombok.accessors.fluent = [true | false] (defecto: false) Si se establece a true, los get y set que se generen no se les añade un prefijo con el estándar get, is o set; en su lugar, los métodos usaran los mismos nombres que las propiedades (sin prefijos). *
      lombok.accessors.prefix += un prefijo de la propiedad (defecto: lista vacía) Esto es una lista de propiedades; las entradas se pueden añadir con el operador +=. Los prefijos heredados de archivos de configuración padres se pueden eliminar con el operador -=. Lombok descartará cualquier coincidencia del prefijo de la propiedad para determinar el nombre de set/get a generar. Por ejemplo, si uno de los prefijos de la lista en esta configuración es m, entonces mAlgo resultará en un get llamado getAlgo(), y no getMAlgo(). *
      lombok.getter.noIsPrefix = [true | false] (defecto: false) Si se establece a true, los get generados por propiedades boolean usarán el prefijo get en lugar del prefijo por defecto is, y cualquier código generado que llame a get, como @ToString, también usará get en lugar de is.
      lombok.setter.flagUsage = [warning | error] (defecto: no establecido) Lombok marcará cualquier uso de @Setter como un warning o un error si se configura.
      lombok.getter.flagUsage = [warning | error] (defecto: no establecido) Lombok marcará cualquier uso de @Getter como un warning o un error si se configura.
      lombok.copyableAnnotations = [Una lista de tipos completamente cualificados] (defecto: lista vacía) Lombok copiará cualquiera de estas anotaciones de la propiedad al parámetro del set, y al método get. Lombok ya trae por defecto una serie de anotaciones que se sabe que son susceptibles de ser copiadas : Todas las anotaciones de tipo no-null/nullables.

      * : Una cadena de parámetros configurada explícitamente de una anotación @Accessors tendrá precedencia sobre esta configuración.
